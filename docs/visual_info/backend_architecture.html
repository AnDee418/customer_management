📋 このシステムについて

このシステムは、顧客情報を安全に管理し、外部システムと連携するための基盤です。
ランサムウェア被害の教訓から、セキュリティ最優先で設計されています。

主要な特徴： リアルタイム更新 ⚡ | 強固なセキュリティ 🔒 | マイクロサービス構成 🔗

💡 ポイント
接続の仕組み：
① BFF → Supabase: 直接接続（ユーザー操作時）
② FastAPI → BFF → Supabase: 外部連携時は BFF の内部API を経由

各層が独立しているため、一箇所が攻撃を受けても他の部分は守られます。
これが「マイクロサービス」の考え方です。

特徴: FastAPIはSupabaseに直接接続せず、必ずBFFの内部APIを経由。セキュリティ強化

🎯 なぜこのような設計？
  • セキュリティ境界の明確化: 外部システムはSupabaseに直接アクセスできない
  • 認証の一元管理: BFFで全ての認証・認可を統一的に処理
  • データ整合性: 顧客IDの解決などのビジネスロジックをBFFに集約
  • 監査ログ: 全ての書き込みがBFFを経由するため、記録漏れがない

🔄 データが流れる仕組み

パターン1: 営業担当が顧客情報を登録する場合

営業担当が画面で顧客情報を入力

名前、連絡先、住所などを入力して「登録」ボタンを押す

BFFが認証チェック

「このユーザーは本当にログインしているか？」を確認

入力データの検証

「電話番号の形式は正しいか？」などをチェック

データベースに保存

Supabaseに安全に保存。自動的に所有者情報も記録

監査ログに記録

「誰が、いつ、何をしたか」を記録（後から確認できる）

画面に完了メッセージ

「登録が完了しました」と表示される

パターン2: 外部システムから発注情報が届く場合

外部システムがWebhookで通知

「新しい発注が入りました！」とリアルタイムで知らせてくる

FastAPIが署名を検証

「本当に正規のシステムからの通知か？」を確認（偽物を弾く）

重複チェック

「この通知は前にも受け取ってない？」を確認

ジョブとして記録

処理の履歴を残す（失敗した場合に再試行できる）

顧客IDを特定

外部の顧客コードから、内部の顧客IDに変換

データベースに保存

発注情報を顧客に紐づけて保存

リアルタイム更新

見ている画面に即座に反映（3秒以内）

🔒 セキュリティ対策

🎫JWT認証

デジタルな「入館証」で正規ユーザーを確認

  • 偽造できない電子署名
  • 有効期限付き
  • 自動更新

🚧IP制限

特定の場所からしかアクセスできない

  • 社内ネットワークのみ許可
  • 不正アクセスを自動ブロック
  • リアルタイムログ記録

⏱️レート制限

短時間に大量アクセスを防ぐ

  • 1分間に100回まで
  • 攻撃を自動検知
  • 一時的にブロック

✅入力検証

不正なデータを事前にブロック

  • 形式チェック
  • サイズ制限
  • 危険な文字を除去

🔐データ暗号化

個人情報を暗号化して保護

  • 保存時に自動暗号化
  • 通信時はHTTPS
  • PII自動マスキング

📝監査ログ

すべての操作を記録

  • 誰が何をしたか記録
  • 変更前後の比較
  • 改ざん不可能

📖 用語解説

用語 わかりやすい説明
BFF 「Backend for Frontend」の略。画面専用の中継サーバー
JWT 偽造できないデジタル証明書。本人確認に使う
Webhook 「何か起きたら教えてね」という仕組み。リアルタイム通知
RLS 「Row Level Security」。データベースで自分のデータだけ見せる
PII 「個人を特定できる情報」。メールや電話番号など
M2M 「Machine to Machine」。システム同士の自動連携
Zod データの形式をチェックするツール（不正データを弾く）
サーキットブレーカー 連続エラー時に自動で処理を止める「安全装置」
